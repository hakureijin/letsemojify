## Current project

**Emoji 进化志 · The Evolution of Emoji** — a single-page bilingual (zh/en) scrollytelling web feature about emoji as a cultural artifact, deployed on port `7777`.

**Hero** (`components/Hero.tsx` + `components/hero/EmojiField.tsx`) — flat cream background with an **interactive emoji wallpaper**: ~275 emojis ordered chronologically by era (1999→2026, left→right / top→bottom), positioned by the pure `lib/hero-emoji-layout.ts` helper (deterministic mulberry32 jitter, central safe-ellipse drop, opacity ramp). Layout profiles: desktop `22×14` cells / base 22 px glyph, mobile `12×16` / 18 px, short-viewport `16×8` / 18 px; when surviving cells exceed the source list, layout wraps modulo so the wallpaper stays dense (each glyph averages ~1.7 appearances at 1440×900). Source list `lib/hero-emoji-timeline.ts` blends chapter-01 `highlightEmojis` with curated era-appropriate supplements (159 unique chars). **Always-on drift**: every emoji button sits inside a `<span.hero-emoji-drift>` wrapper running a 5-keyframe CSS animation; per-emoji `--drift-duration` (6–12 s) and negative `--drift-delay` (−5..0 s) are seeded from index so the field looks asynchronous from the first frame and never visually resyncs. GSAP (free core + `@gsap/react`) drives a **magnetic-push hover** (single rAF-throttled `mousemove` → per-emoji `quickTo`, 100 px radius, `elastic.out(1, 0.5)` spring-back on `mouseleave`) and a **click-to-enlarge toggle** (active emojis `scale: 3` with `back.out(1.7)`, neighbours within 120 px receive summed push vectors from all active emojis). Esc collapses all popped emojis. Drift sits on the wrapper, GSAP transforms the inner button — independent transform targets, so they never fight. Hover handler freezes while any emoji is active to avoid conflicting transforms. `prefers-reduced-motion` skips the hover listener, makes click changes instant, AND disables the drift animation; touch-only devices (`matchMedia('(hover: hover) and (pointer: fine)')`) skip magnetic push but still get click + drift. A `.hero-vignette::after` radial-gradient overlay washes the centre with `var(--bg)` so the title / eyebrow / subtitle / scroll cue stay at WCAG-AA contrast despite the dense backdrop. Each emoji is a `<button>` with `aria-pressed` + per-locale `aria-label` template (`hero.enlarge` / `hero.shrink` in `messages/{zh,en}.json`). Hero title currently reads "How We Emojify: Trends and Patterns in Emoji Usage" (same string in both locales).

The site has two chapter sections, in time order:

1. §01 — Three stacked interactive charts (no chapter eyebrow / title / intro block; the section is just `<Section id="ch01">` anchor for `TopNav`'s `01` link, then the charts). (a) **Cumulative-growth chart** (`components/chapter-01/CumulativeChart.tsx`): each Unicode version is an **emoji medallion** (the version's first highlight emoji on a circle), with decade-anchor years (1999/2010/2015/2020/2024/2026) styled as larger flagship markers. Three **range-zoom buttons** (`全部 / 2015→ / 2020→`) narrow the x-domain so the dense modern era spreads out. Tooltips show year + version + +new + cumulative + growth-percent + narrative + source on hover/Tab+Enter/tap; clip-path keeps the line/area visually continuous across range changes. Two **version-diff dropdowns** (`从 / 到`) beneath the chart let readers pick any two contributing versions; the chosen markers gain bold accent strokes and small A/B chips, and a card below shows added count, growth %, year span, and a chronological strip of representative new emojis for the intermediate versions. Pure `computeVersionDiff` exported from the same component file and covered by `tests/logic/version-diff.test.ts`. The chart consumes `ch01.timeline.*.narrative` i18n keys for the tooltip per-version narratives. (b) **Category treemap** (`components/chapter-01/CategoryTreemap.tsx`) drives a Unicode-version time slider (snap-points at Emoji 6.0/2010 → latest) over a 9-tile treemap of CLDR groups — Smileys & Emotion, People & Body, Animals & Nature, Food & Drink, Travel & Places, Activities, Objects, Symbols, Flags — with play/pause, Framer Motion rect transitions, tooltips, and a screen-reader `<details>` summary. Each tile is anchored by a single hero emoji glyph sized to ~78% of the tile's short axis (capped at 96 px); the group name + count + percent live in a slim bottom band with a subtle dark scrim for legibility. The bottom band uses a 3-tier layout chosen per tile from its width AND an estimated label width (CJK glyphs ~1 em, Latin uppercase bold ~0.95 em per char): **wide tile + label fits** → single row (label left, count right); **narrow tile but label fits** → stack label-on-top, count-below in a 42 px band; **label can't fit** (e.g. "Smileys & Emotion" in a 122 px tile at early Unicode versions) → drop the label entirely and centre the count alone. Result: zero overflow at all 17 slider frames in both locales, verified by a Playwright scrub. Data lives in `data/chapter-01-categories.json`, regenerated by `scripts/build-emoji-categories.mjs` from `unicode.org/Public/emoji/latest/emoji-test.txt`. (c) **Variant-complexity Sankey** (`components/chapter-01/VariantSankey.tsx`) decomposes the latest catalogue snapshot (Emoji 17.0, 3,944 entries) by 8 variant mechanisms — base / skin-tone / multi-skin-tone / ZWJ family / ZWJ role / ZWJ other / hair-style / direction-flipped — and flows each mechanism into the same 9 CLDR groups. The classifier (`classifyEmojiVariant`, exported from the build script and unit-tested) inspects the codepoint sequence: ZWJ presence, hair / arrow / person codepoints, and skin-tone modifier count. Default rendering is clean colored ribbons; the active flow (hover / focus / pin) unpacks emoji beads along its Bézier curve via the local `sampleSankeyCurve()` helper, with bead count scaling as `Math.log2(value)*0.85` (1–10 beads) and bead size as `clamp(13, width*0.55, 22)`. Beads are `aria-hidden`; the flow's existing `aria-label` carries the screen-reader narration. Data lives in `data/chapter-01-variants.json`, produced by the same build script in a second output pass.
2. §02 谁能成为 emoji / Who Gets In — Unicode proposal pipeline + **minimalist criteria cards** (no left-edge stripes; tone shown via a small uppercase tag inside each card) + 4 case cards + **interactive pan/zoom cultural-origins world map** (`components/chapter-02/OriginMap.tsx`). 41 origin pins span 23 countries from every populated continent, anchored at city-level coordinates so multi-pin countries (CN×3, JP×3, IN×4, MX×4, FR×3, IT×2, DE×2, TR×2, AU×2) only resolve visually after zooming in. The map supports mouse-wheel zoom toward cursor (1×–6× clamped), drag-to-pan, two-finger pinch on touch, and keyboard navigation on the focused SVG (`+ / − / 0` and arrow keys). A floating button group (`+ / − / ⟲`) and a bottom-left zoom-level indicator round out the controls. Pins are positioned in screen space (`cx*k + tx, cy*k + ty`) so their visible size stays constant while the map scales; off-viewport pins are skipped from render via an `inViewport` filter. Drag uses mouse + touch events with document-level `mousemove/mouseup` (or `touchmove/touchend`) listeners attached on press — this pattern lets drag start anywhere (including over a pin's hit area) without the `setPointerCapture` click-retargeting that previously prevented panning, and the browser's own movement-threshold click suppression keeps pin-click and map-drag from conflicting (with `justDraggedRef` as a belt-and-braces guard). `clampZoom` allows a generous ±30% pan margin even at `k=1` so the drag is perceivable at default zoom.

Earlier versions of the page included §03 (Always On, generational phone-engagement stats) and §04 (Beyond the Screen, AI/AR/brand/interface case gallery). Both were removed on 2026-05-21 to tighten the narrative around the catalogue itself. §01 also originally opened with a horizontal pin-and-translate emoji-version timeline (Framer Motion `useScroll`) plus a decade-jump nav row; both were removed on 2026-05-26 so §01 contains only the three charts above. Components, data, types, tests, and i18n keys for the removed features are gone from the working tree — git history retains them (see commits `538cd07` and `58c5ea6` for the §01 cleanup).

Visual register is intentionally playful (gradient backgrounds, rounded cards, emoji-led big type) while content discipline is strict — every quantitative claim has a source URL and surfaces in the footer bibliography.

Tech stack: Next.js 15 App Router · React 19 · TypeScript · Tailwind v4 · next-intl · Framer Motion (charts) · GSAP + `@gsap/react` (hero only) · d3-* + world-atlas · Vitest.

## Relationship to the previous Figure 4.17 work

The prior brainstorm in this repo was a single-page mockup for Figure 4.17 of the paper (a bump chart of "Top emojis on X.com: 2023-2026"). Per the new spec's §1, this current project is **intentionally decoupled** from that work — it occupies the same port 7777 but is not a continuation. The Figure 4.17 mockup artifacts still live under `.superpowers/brainstorm/1421938-1777524397/` and are tracked in git, in case they're useful as a style reference.

Useful research notes carried over from the Figure 4.17 brainstorm (still accurate as background):

- **`😂` Face with Tears of Joy** had long-term dominance on Twitter through the 2010s.
- **`😭` Loudly Crying Face** challenged/overtook `😂` around the COVID pandemic.
- **`🥺` Pleading Face** declined after its pandemic-era rise.
- These shifts informed the now-removed §03's "same emoji, different meaning by generation" block — kept here for context if anyone ever revives that section.

Data sources investigated in the prior brainstorm (some used here, some not):

- Emojipedia · `https://emojipedia.org/emoji-versions` — authoritative for per-version emoji counts; **heavily used in §01**.
- X.com / Twitter API — full-archive search and counts. Not used in current project (would need paid access).
- Emojitracker · `https://emojitracker.com/` — real-time popularity; not used.

## Open follow-ups

- **§02 rejected case** — the spec called for a verified rejected Unicode proposal. The current data ships with a "deity category" policy-based rejection (citing the Unicode exclusion factors page) because no specific L2 PDF for a rejected proposal could be confirmed during the implementation pass. If a real rejected proposal can be found in the L2 archive, swap it in.
- **A11y verification in a real browser** — code-level fixes are in place (MotionConfig, prefers-reduced-motion CSS, focus-visible rings, 44 px touch targets on chart markers, role/aria-expanded/aria-live on tooltips), but a Lighthouse / axe-core run in an actual browser was not performed during the implementation session.
- **Content depth** — §02 has 4 cases / 41 origin pins across 23 countries. Adding more pins is data-only (`data/chapter-02.json` + parallel label keys in `messages/{zh,en}.json`). The pan/zoom interaction was sized to handle this density — at default zoom dense clusters (Europe, East Asia, Mexico) overlap visually; users zoom in to disambiguate.
- **Stale `cumulative` array** — the §01 chart now derives the running total from `timeline` directly. The `cumulative` field in `data/chapter-01.json` and the `Chapter01Data['cumulative']` type are no longer read by any component. They're still validated by `tests/data/chapter-01.test.ts` for shape consistency, but could be removed in a future cleanup pass.
- **Stale `decadeIndex` array** — same situation as `cumulative`. The field was consumed by the now-deleted `DecadeIndex.tsx` decade-jump nav. Still present in `data/chapter-01.json` and `Chapter01Data['decadeIndex']`; safe to delete in a cleanup pass.
- **Treemap pre-2010 era is omitted** — `chapter-01-categories.json` starts at Unicode 6.0 (2010) because Unicode CLDR's group taxonomy doesn't categorise the pre-Unicode DoCoMo 1999 / iPhone JP 2007 sets. The slider caption discloses this. The cumulative line chart above the treemap still shows 1999 — they're complementary, not duplicate, views.
- **Treemap totals vs. timeline counts diverge** — `chapter-01-categories.json` counts *every fully-qualified emoji* in `emoji-test.txt` (singles + sequences + skin-tone variants), so the Emoji 17.0 total (~3,944) is larger than the sum of `newEmojiCount` in `chapter-01.json` (which uses Emojipedia's headline "new emoji" numbers per version). Both are correct for their respective questions — cumulative line chart answers "how many emoji were added at this milestone (per Emojipedia)?"; treemap answers "what's the *total* breakdown per category right now (per Unicode CLDR)?".
- **Emoji 17.0 stand-in glyphs** — Emoji 17.0 shipped Sept 2025 with new codepoints (🫪 🫯 🫍 🪊 etc.) that most system fonts won't have until 2026 platform rollouts complete. The chart medallion currently shows visually-similar pre-existing emojis (🫨 💥 🐳 🎺) with a stand-in disclosure in the tooltip narrative. When system-font support is widespread, the `highlightEmojis` array in `data/chapter-01.json` can be reverted to the canonical 17.0 codepoints.
- **Emoji 18.0 codepoints are provisional** — final glyphs may shift between now (May 2026) and the Sept 2026 release. Same stand-in pattern as 17.0; expected to need updates when 18.0 is finalised.
- **Treemap data refresh** — when Unicode 18.0 ships (expected Sept 2026), re-run `node scripts/build-emoji-categories.mjs` to regenerate both `data/chapter-01-categories.json` and `data/chapter-01-variants.json`, then add an `18.0` entry to the `VERSION_RELEASE` map inside the script.
- **Treemap label-width heuristic** — the per-tile label-fit decision uses a character-count × per-char-width estimate (CJK ~11 px, Latin uppercase bold ~9.5 px at the chart's 10 px font + 0.06 em tracking). If a new locale or font is added with very different glyph metrics, the per-char constants need to be re-tuned. Verified empirically by Playwright scrubbing all 17 frames and confirming `text.getBoundingClientRect().width` never exceeds `tile.width - 4`.
- **Sankey classifier ambiguity** — `classifyEmojiVariant` assigns each emoji to exactly one mechanism using a prioritised rule order (hair-style > direction-flipped > zwj-family > multi-skin-tone > zwj-role > zwj-other > skin-tone > base). Trade-offs documented in the build script header and disclosed in the Sankey caption. Edge cases the rule order covers: a hair-style ZWJ with skin tone → hair-style (hair is the visual story); a couple with 2 skin tones → zwj-family (relationship trumps tones); a multi-skin-tone handshake (1FAF1 1F3FB 200D 1FAF2 1F3FC) → multi-skin-tone (no PERSON_CODEPOINTS member, falls through to skin-count rule).

## Conventions for future edits

- **Adding a data row**: add to `data/chapter-XX.json` with a `source` block, then add the corresponding string keys to BOTH `messages/zh.json` and `messages/en.json`. The i18n parity test enforces this.
- **Adding a new section / chapter**: extend the types under `types/`, add a `data/chapter-XX.json`, write the components under `components/chapter-XX/`, wire into `app/[locale]/page.tsx`. Follow the existing per-chapter accent-color pattern (each chapter has one primary CSS variable accent).
- **Animations**: chart animations use Framer Motion gated by the global `<MotionConfig reducedMotion="user">`. The hero wallpaper specifically uses GSAP core (`gsap.quickTo` for the magnetic-push hot path, `gsap.to` for click toggles) with `@gsap/react`'s `useGSAP` for cleanup, plus CSS `@keyframes` for the always-on ambient drift; this combination was chosen so the three motion sources don't fight over `transform`. The CSS `@media (prefers-reduced-motion: reduce)` block in `globals.css` is the safety net for anything outside those two libraries. When adding new charts prefer Framer Motion to keep the codebase consistent; reserve GSAP for hero-style "page furniture" where it already pulls its weight.
- **Interactive chart marker pattern**: `CumulativeChart`, `OriginMap`, and `DayInLife` share the marker shape — `useState` for `activeId`/`pinnedId`, `useEffect` for outside-click + Escape, 44 px invisible hit areas, `role="button" tabIndex={0}`, hover/focus/blur for transient state, click + Enter/Space for pin/unpin. New interactive charts should follow this same shape for consistency.
- **Pan/zoom interaction pattern (OriginMap-only for now)**: maps use a separate `ZoomState { k, x, y }` rendered via two SVG groups — country paths inside `<g transform="translate scale">` so they scale, pins drawn at screen-space coords (`cx*k + tx, cy*k + ty`) so they don't. Mouse drag attaches `mousemove/mouseup` listeners to `document` on press (NOT `setPointerCapture`, which retargets click events and breaks pin-clicks); touch attaches `touchmove/touchend` listeners and detects 2-finger pinch by tracking simultaneous touches. `clampZoom` always allows ≥30% pan margin so dragging feels responsive even at `k=1`. Wheel zooms toward cursor via `addEventListener('wheel', …, { passive: false })` so `preventDefault()` actually suppresses page-scroll. Keyboard `+ / − / 0 / Arrow*` works on the focused SVG. Reuse this pattern if any future chart needs spatial pan/zoom.
- **i18n with fully-qualified keys**: when a component's `useTranslations()` has a namespace prefix (e.g. `useTranslations('ch01.chart')`) but it also needs to resolve a fully-qualified key from data (e.g. `node.narrativeKey = 'ch01.timeline.x.narrative'`), declare a second un-namespaced `useTranslations()` for the fully-qualified path. Otherwise the namespace double-prefixes and produces `MISSING_MESSAGE` errors. See `CumulativeChart.tsx` and `OriginMap.tsx`.
- **Stand-in emojis for unsupported codepoints**: if a data row references emojis that aren't yet in mainstream system fonts (e.g. Emoji 17.0 / 18.0 codepoints), use a visually-similar pre-existing emoji in `highlightEmojis` and disclose the substitution in the narrative text. Keep the source URL pointing at the canonical version page.
- **Commits**: project uses git on `master` branch. Commit messages follow `<type>: <imperative>` form (e.g., `feat:`, `fix:`, `data:`, `test:`, `chore:`).

## Known dependency quirks

- **Node 18.20** is enough to run, but `@tailwindcss/oxide` prefers Node 20+. The native binding `@tailwindcss/oxide-linux-x64-gnu` had to be installed explicitly (it's now a `devDependency`) to work around an npm optional-deps bug. If you upgrade to Node 20+, this manual dep can be removed.
- **`pnpm` is not installed** on the dev host. Project scripts use `npm`. The package-lock.json is committed.
- Production build uses **Turbopack** (`next build --turbopack`). Dev too.
